如果 Redis 主节点刚加锁成功，但还没来得及同步给从节点，主节点就宕机了，这时从节点被提升为新主节点。其他客户端可能又能重新加锁成功，于是就可能出现 **两个客户端同时认为自己持有锁** 的情况。

**跳表底层数据结构？跳表和b+树区别？**

跳表的底层结构是多层有序链表。每个节点除了保存 key、value 之外，还会维护一个 forward 指针数组，表示它在不同层级上的下一个节点。最底层链表包含所有数据，上层链表相当于索引，用来加速查找。查找时从最高层开始，能往右就往右，不能往右就下降一层，最终在底层找到目标。跳表通过随机层数来维持概率平衡，查询、插入、删除的平均时间复杂度都是 O(log n)。

跳表和 B+ 树的主要区别在于：跳表是链表加多级索引，B+ 树是多叉平衡搜索树；跳表通过随机层数维持平衡，B+ 树通过节点分裂和合并维持平衡；跳表实现简单，适合内存场景，比如 Redis ZSet；B+ 树节点可以和磁盘页对齐，树高低，磁盘 IO 少，所以更适合数据库和文件系统这类磁盘索引场景。两者都支持范围查询，复杂度一般都是 O(log n + k)。

**GC**

Java GC 是 JVM 自动管理堆内存的过程，主要作用是回收不再被使用的对象，释放内存，避免内存泄漏和内存溢出。

**分布式服务 对接口做限流 实现思路？**

分布式服务对接口限流，核心是控制单位时间内进入系统的请求量，避免突发流量打垮服务。限流可以做在网关层、应用层和资源层。网关层适合做全局 QPS、IP、用户、租户限流；应用层适合根据业务参数做细粒度限流；资源层用于保护数据库、Redis、线程池、第三方接口等下游资源。

常见限流算法有固定窗口、滑动窗口、漏桶和令牌桶。固定窗口实现简单但有临界突刺问题；滑动窗口更平滑；漏桶可以稳定流出流量；令牌桶既能限制平均速率，又允许一定突发流量，所以实际使用比较多。

如果是单机限流，可以用 Guava RateLimiter、Semaphore、Sentinel 等；如果是分布式限流，需要多个服务实例共享限流状态，常见方案是网关统一限流，或者使用 Redis + Lua 实现全局计数器、滑动窗口或令牌桶。Lua 的作用是保证计数、过期、判断这些操作的原子性。

实际项目中，我会优先在网关做粗粒度全局限流，在应用层做用户、接口、热点参数等细粒度限流，并配合熔断、降级、缓存和 MQ 削峰。限流触发后可以返回 429、友好提示、降级结果，或者把请求异步排队处理。



**redis淘汰策略详细讲一下？**

Redis 淘汰策略是在设置了 `maxmemory` 后，内存达到上限并且有新的写入请求时触发的。它决定 Redis 是拒绝写入，还是删除一部分 key 来腾出空间。

常见策略可以分为三类：第一类是 `noeviction`，内存满了不淘汰，直接拒绝写请求；第二类是 `volatile` 系列，只从设置了过期时间的 key 中淘汰，比如 `volatile-lru`、`volatile-lfu`、`volatile-random`、`volatile-ttl`；第三类是 `allkeys` 系列，从所有 key 中淘汰，比如 `allkeys-lru`、`allkeys-lfu`、`allkeys-random`。

LRU 表示最近最少使用，优先淘汰最近很久没访问的 key；LFU 表示最不经常使用，优先淘汰访问频率最低的 key；TTL 策略会优先淘汰剩余过期时间最短的 key。需要注意的是，Redis 的 LRU 和 LFU 都不是完全精确实现，而是通过随机采样做近似淘汰，采样数量由 `maxmemory-samples` 控制。

实际项目中，如果 Redis 纯粹作为缓存，常用 `allkeys-lru` 或 `allkeys-lfu`；如果 Redis 中既有缓存又有重要数据，可以使用 `volatile-lru` 或 `volatile-lfu`，只让设置了过期时间的缓存 key 被淘汰；如果数据不能丢，就使用 `noeviction`，让写请求失败，由业务处理。

**Java反射理解，Aop设计思路？**

Java 反射是指程序在运行时动态获取类的信息，并且可以动态创建对象、调用方法、访问字段的一种机制。

把与业务无关但多个模块都会用到的公共逻辑抽取出来，通过切面统一增强，避免侵入业务代码。

Java 反射是运行时动态获取类信息、创建对象、访问字段、调用方法和读取注解的机制。它的核心入口是 Class 对象，常用 API 包括 Constructor、Field、Method。反射广泛应用在 Spring IOC、依赖注入、MyBatis、JSON 序列化、JUnit 等框架中。它的优点是灵活、解耦，缺点是性能比直接调用低，而且可能破坏封装，很多错误要到运行时才暴露。

AOP 是面向切面编程，核心思想是把日志、事务、权限、限流、监控这类横切逻辑从业务代码中抽离出来，通过代理统一增强。Spring AOP 中有连接点、切点、通知、切面、织入等概念。底层实现主要依赖动态代理，如果目标类实现了接口，可以用 JDK 动态代理；如果没有接口，可以用 CGLIB 生成目标类子类进行代理。

AOP 的设计思路是：启动时扫描切面，根据切点表达式判断哪些 Bean 需要增强，然后为这些 Bean 创建代理对象。业务调用时实际调用的是代理对象，代理对象在执行目标方法前后插入增强逻辑，比如开启事务、记录日志、权限校验等。反射为 AOP 提供了运行时方法调用能力，而动态代理则是 AOP 实现方法增强的关键机制。

**AOP失效**

AOP 失效的核心原因是调用没有经过 Spring 代理对象。Spring AOP 是基于动态代理实现的，只有通过代理对象调用方法，切面逻辑才会执行。

常见失效场景包括：第一，同一个类内部方法自调用，比如 `this.method()`，不会经过代理对象；第二，目标对象不是 Spring 容器管理的 Bean，而是自己 new 出来的；第三，方法是 private、final，或者类是 final，代理无法增强；第四，切点表达式没有匹配到目标方法；第五，事务场景下异常被 catch 掉没有抛出，或者抛出的是受检异常但没有配置 `rollbackFor`；第六，多线程场景下事务上下文不能自动传播，因为事务信息通常存在 ThreadLocal 中。

所以排查 AOP 是否失效时，我一般会先看这个对象是不是 Spring Bean，再看调用是否经过代理对象，然后检查方法修饰符、切点表达式、异常处理和事务传播配置。

**元空间OOM**

元空间 OOM 对应的错误是 `java.lang.OutOfMemoryError: Metaspace`。JDK 8 之后，永久代被移除，方法区主要由元空间实现，元空间使用的是本地内存，主要存放类元信息、方法信息、字段信息、运行时常量池相关信息以及类加载器相关数据。

元空间 OOM 的常见原因是类加载过多、动态生成大量代理类，或者 ClassLoader 泄漏。比如反复热部署、插件系统、自定义类加载器、CGLIB 动态代理、脚本引擎等场景，如果旧的 ClassLoader 无法被回收，它加载的类也无法卸载，元空间就会持续增长。

排查时可以看 GC 日志和类加载数量，比如用 `jstat -class` 观察 Loaded 和 Unloaded 的变化，也可以通过 dump 分析是否存在大量 ClassLoader 或动态代理类。解决上，一方面可以合理设置 `-XX:MetaspaceSize` 和 `-XX:MaxMetaspaceSize`，另一方面更重要的是排查类加载器泄漏，关闭线程池、清理 ThreadLocal、注销 JDBC Driver、释放自定义 ClassLoader，避免无限动态生成类。

**对比一下 CMS 和 G1**

CMS 和 G1 都是以降低 GC 停顿为目标的垃圾回收器，但 CMS 更偏老年代并发回收，G1 是面向整堆的分区化回收器，目标是可预测停顿时间。

```
CMS 和 G1 都是为了降低 GC 停顿而设计的垃圾回收器。CMS 主要作用在老年代，通常和 ParNew 搭配使用。它采用标记-清除算法，流程包括初始标记、并发标记、重新标记和并发清除，其中初始标记和重新标记会 STW，并发标记和并发清除可以和用户线程一起执行，所以停顿时间比传统 Full GC 短。
但是 CMS 的问题也比较明显，因为它是标记-清除算法，会产生内存碎片；并发阶段用户线程还在运行，会产生浮动垃圾；如果老年代空间不足或者回收跟不上分配，就可能发生 Concurrent Mode Failure，退化成 Full GC，停顿时间会很长。
G1 则是把堆划分成多个 Region，逻辑上仍然有新生代和老年代，但物理上不要求连续。G1 会通过并发标记统计各个 Region 的垃圾比例，然后优先回收收益最高的 Region，所以叫 Garbage First。它采用复制和整理的方式回收 Region，碎片问题比 CMS 更少，并且可以通过 -XX:MaxGCPauseMillis 设置期望停顿时间，因此停顿时间更可预测。
简单来说，CMS 是传统分代模型下的老年代并发标记清除收集器，优点是降低停顿，缺点是碎片和浮动垃圾；G1 是 Region 化的整堆收集器，优点是可预测停顿和更少碎片，更适合大堆服务端应用。
```

**G1 回收器的STW时间是如何控制的？**

把堆拆成很多 Region，每次不一定回收整个堆，而是根据停顿时间目标，选择一部分“回收收益最高”的 Region 来回收。

**详细解释 MySQL 的 MVCC机制是如何实现的？**

MVCC，全称 Multi-Version Concurrency Control，多版本并发控制。它的核心思想是：同一条数据在数据库中可以存在多个历史版本，读操作通过读取合适的历史版本来避免和写操作互相阻塞，从而提高并发性能。

1. 隐藏字段
2. undo log 版本链
3. Read View 读视图

在 InnoDB 的 MVCC 中，Undo Log 和 Read View 是两个核心组件。Undo Log 负责保存记录的历史版本，每行记录通过隐藏字段 `DB_ROLL_PTR` 指向 Undo Log 中的旧版本，从而形成一条版本链；而 `DB_TRX_ID` 记录最近修改该版本的事务 ID。

Read View 负责判断版本可见性。它会记录创建读视图时系统中活跃事务的 ID 列表、最小活跃事务 ID、下一个事务 ID 以及当前事务 ID。事务读取数据时，会拿记录版本上的 `DB_TRX_ID` 和 Read View 做比较，判断该版本是否对当前事务可见。

如果当前最新版本不可见，就通过 `DB_ROLL_PTR` 去 Undo Log 中找上一个历史版本，继续判断，直到找到一个可见版本。

所以可以总结为：Undo Log 解决“有没有历史版本可读”的问题，Read View 解决“应该读哪个历史版本”的问题。

**RC（读已提交）和 RR（可重复读）隔离级别下，Read View 的生成时机有什么不同？**

RC 是每次快照读都会生成新的 Read View；RR 是事务中第一次快照读生成 Read View，后续复用同一个 Read View。

**在 RR 隔离级别下，MySQL 是如何解决幻读问题的？**

快照读通过 MVCC 解决幻读，当前读通过 Next-Key Lock 解决幻读。

**为什么普通的快照读不需要加锁？**

它读的不是正在被修改的“最新数据”，而是根据 Read View 读取一个对当前事务可见的历史版本。

**事务提交时，Redo Log 和 Binlog 的写入顺序是怎样的？（两阶段提交）**

```
1. 写 Redo Log，并标记为 prepare 状态
2. 写 Binlog
3. 提交事务，把 Redo Log 标记为 commit 状态
```

**如果 Redo Log 写完了但 Binlog 没写完时发生了宕机，MySQL 会如何处理？**

1. InnoDB 扫描 Redo Log，发现这个事务处于 prepare 状态
2. MySQL 检查 Binlog 中是否有这个事务的完整记录
3. 如果 Binlog 没有写完，或者没有完整事务记录
4. 说明这个事务还没有真正提交成功
5. InnoDB 会回滚这个事务

**如果 Redis 的 Lua 脚本执行时间过长（例如包含复杂循环或IO操作），会产生什么后果？应该如何避免？**

```
Redis 是单线程执行命令的，Lua 脚本执行期间会阻塞 Redis 主线程，其他命令都要等待。
```

避免方式是：Lua 脚本只做短小的原子逻辑，比如分布式锁释放、限流计数、库存扣减等；不要在脚本中做复杂循环、大量 key 扫描、大集合遍历或耗时计算；批量任务要拆成小批次；复杂业务逻辑放在应用层，Lua 只做最后的原子判断和写入。同时要监控 Redis slowlog、EVAL 耗时、客户端超时和主从延迟。

简单总结就是：Redis Lua 的优势是原子性，风险是阻塞主线程，所以脚本必须短、快、可控。

**快速排序模板**

```java
public class QuickSort {
    public static void quickSort(int[] arr, int left, int right) {
        if (left >= right) return;
        int pivot = arr[left + (right - left) / 2];
        int i = left, j = right;
        while (i <= j) {
            while (arr[i] < pivot) i++;
            while (arr[j] > pivot) j--;
            if (i <= j) {
                int tmp = arr[i];
                arr[i] = arr[j];
                arr[j] = tmp;
                i++;
                j--;
            }
        }
        quickSort(arr, left, j);
        quickSort(arr, i, right);
    }
}
```



**索引失效的原因**

MySQL 索引失效的常见原因主要有：
 第一，对索引字段使用函数、表达式计算或发生隐式类型转换；
 第二，联合索引不满足最左前缀原则，或者范围查询导致后面的字段无法继续使用索引；
 第三，LIKE 使用左模糊匹配，例如 `%abc` 或 `%abc%`；
 第四，OR 条件中存在未建索引字段，或者使用 `!=`、`<>`、`NOT IN`、`NOT LIKE` 等选择性较差的条件；
 第五，查询命中数据量过大、字段区分度低、回表成本高，优化器认为全表扫描更划算；
 第六，ORDER BY、GROUP BY 的字段顺序和索引顺序不一致，也可能导致索引无法用于排序。

**Redis CAS乐观锁是怎么做的？**

Redis实现CAS乐观锁依赖**Check and Set**模型，核心指令是`WATCH`。原理可以概括为：

> `WATCH`监控一个或多个key，如果在`MULTI`到`EXEC`这段事务窗口期内，被监控的key被其他客户端修改，则**整个事务被放弃**，返回nil。

| 维度           | WATCH+MULTI                | Lua脚本              | Redlock分布式锁        |
| :------------- | :------------------------- | :------------------- | :--------------------- |
| **原子性保证** | 乐观锁，冲突时放弃事务     | 强原子，脚本整体执行 | 悲观锁，加阻塞         |
| **并发性能**   | 高（无锁等待，但需要重试） | 最高（单次交互完成） | 较低（等待锁释放）     |
| **实现复杂度** | 中等，需处理重试逻辑       | 低，但需掌握Lua语法  | 低，但需管理锁超时     |
| **适用场景**   | 通用乐观更新               | 高并发CAS校验        | 跨服务互斥、长事务     |
| **典型应用**   | 计数器、库存扣减           | 秒杀库存、配置更新   | 定时任务防重、分布式锁 |
